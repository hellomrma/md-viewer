## 이 가이드는 이런 분을 위한 것입니다

**예상 소요시간**: 처음부터 끝까지 따라하면 약 2-3시간 (실습 포함).

## Part 0 — 큰 그림

## 1. Claude Code와 Claude Cowork — 두 가지 입구

### 1.1 한 줄로 정리

| 이름 | 한 줄 설명 |
| --- | --- |
| **Claude Code** | 터미널(검은 화면)에서 쓰는 AI 비서. 개발자에게 익숙한 환경. |
| **Claude Cowork** | 데스크톱 앱(아이콘 클릭)에서 쓰는 AI 비서. 비개발자도 바로 쓸 수 있음. |

둘 다 **AI가 내 컴퓨터의 파일을 직접 다룬다**는 점은 같습니다. 다른 점은 "어떻게 들어가느냐"입니다.

### 1.2 비유로 이해하기

같은 회사 비서가 두 가지 출입구를 가지고 있다고 생각하세요.

- **Claude Code**: 비서실 정문(터미널). 명령어로 직접 부르는 방식. 자유도 ↑, 진입 장벽 ↑.
- **Claude Cowork**: 비서실 옆문(앱 클릭). 버튼과 메뉴로 부르는 방식. 진입 장벽 ↓, 일부 기능은 더 강력.

### 1.3 무엇이 같고 무엇이 다른가?

| 항목 | Claude Code | Claude Cowork |
| --- | --- | --- |
| 인터페이스 | 터미널 | 데스크톱 앱 |
| 파일 다루기 | ✅ 직접 | ✅ 직접 (폴더 연결 후) |
| Skill | ✅ | ✅ |
| Subagent | ✅ (`.claude/agents`) | ✅ (자동 위임) |
| Slash Command | ✅ | ✅ (Skill처럼 호출) |
| Hook (자동 약속) | ✅ | 제한적 |
| **Artifact (라이브 페이지)** | ❌ | ✅ |
| **Scheduled Task (예약 실행)** | ❌ | ✅ |
| **Connector (슬랙·노션 등)** | MCP로 가능 (수동 설정) | ✅ UI에서 클릭으로 연결 |
| **Plugin Marketplace** | CLI로 가능 | ✅ UI에서 한 번에 설치 |
| 추천 사용자 | 개발자, CLI 익숙 | 모든 사용자, 특히 비개발자 |

### 1.4 핵심 메시지

- **개념(Skill, Subagent, Slash Command)은 거의 동일**합니다. 한 번 익히면 양쪽에서 같이 쓸 수 있습니다.
- **Cowork에만 있는 강력한 기능**(Artifact, Scheduled Task, Connector)이 있어, 일상 업무 자동화에서는 Cowork가 더 편리합니다.
- 둘은 경쟁이 아니라 **상황에 맞춰 선택**하는 도구입니다.

## 2. 어느 쪽으로 시작할까?

### 2.1 결정 트리

```
"나는 터미널(검은 화면) 사용에 익숙한가?"
    ├─ 아니요 / 잘 모르겠다
    │      → Cowork로 시작하세요 (3장으로)
    │
    └─ 네, 어느 정도 익숙합니다
           ├─ "주로 코드를 다루나요?"
           │      ├─ 네 → Claude Code (4장으로)
           │      └─ 아니요 → Cowork도 충분 (3장으로)
           │
           └─ "슬랙·노션·구글캘린더 등 외부 서비스와 연결이 자주 필요한가요?"
                  └─ 네 → Cowork 권장 (Connector 강력)
```

### 2.2 추천 시나리오

| 당신의 일 | 권장 |
| --- | --- |
| 회의록 정리, 보고서 작성, 메일 회신 | **Cowork** |
| 슬랙·노션·구글 드라이브와 자주 일함 | **Cowork** |
| 매주/매일 정해진 시간에 자동 작업 필요 | **Cowork** (Scheduled Task) |
| 코드 작성, 깃허브 PR 작업 | **Claude Code** |
| 터미널/스크립트 작업이 많음 | **Claude Code** |
| 둘 다 쓰고 싶음 | **양쪽 다 설치 가능** (계정 1개로 사용) |

### 2.3 한 가지 좋은 소식

Skill, Subagent, Slash Command는 **두 도구에서 거의 똑같이 동작**합니다. 한 쪽에서 만든 것을 다른 쪽으로 가져가도 대부분 그대로 작동합니다. 그러니 "잘못된 선택을 할까봐"라는 부담 없이 골라도 됩니다.

## 3. Cowork로 시작하기 (비개발자 권장)

### 3.1 Cowork가 뭔가요?

Cowork는 **Claude의 데스크톱 앱**입니다. 다음과 같은 화면에서 일합니다.

- 왼쪽: 대화 목록
- 가운데: 지금 대화창
- 오른쪽: 사이드 위젯 (TodoList, Artifact 미리보기 등)

ChatGPT 데스크톱 앱과 비슷한 느낌이지만, **내 컴퓨터의 파일을 직접 다루고**, **외부 서비스(슬랙·노션 등)와 연결**할 수 있다는 점이 다릅니다.

### 3.2 한 단계씩 따라하기 — 설치부터 첫 대화까지

#### 1단계 — Claude 데스크톱 앱 설치

1. 웹 브라우저에서 https://claude.com/download 접속
2. 본인 OS에 맞는 설치 파일 다운로드 (Mac / Windows)
3. 설치 → 실행 → Anthropic 계정으로 로그인

#### 2단계 — Cowork 모드 진입

앱을 열면 일반 채팅 화면이 보입니다. 화면 어딘가에 **"Cowork"** 또는 **"파일 작업"**, **"폴더 연결"** 같은 버튼이 있습니다(버전에 따라 위치가 다를 수 있습니다). 이를 누르면 Cowork 모드로 들어갑니다.

> 💡 처음 진입 시 **"역할 선택"** 같은 가이드가 뜰 수 있습니다. 본인 직무에 가까운 것을 고르면 추천 Plugin이 자동으로 설치됩니다.

#### 3단계 — 작업할 폴더 연결하기

Cowork에서 가장 먼저 해야 할 일은 **"내 컴퓨터의 어느 폴더에서 작업할지"** 알려주는 것입니다.

1. 새 작업 폴더를 만듭니다 (예: 바탕화면에 `claude-test`).
2. Cowork 대화창에 다음과 같이 입력합니다:
   ```
   바탕화면의 claude-test 폴더에서 작업할게.
   ```
3. Claude가 "폴더 접근 권한이 필요합니다"와 같은 요청을 띄웁니다. **"허용"** 또는 **"폴더 선택"** 클릭.
4. 폴더 선택 창에서 위에서 만든 폴더를 고릅니다.

이제 Cowork는 그 폴더 안에서 파일을 만들고 읽고 고칠 수 있습니다.

#### 4단계 — 첫 대화 해보기

```
hello.txt 파일을 만들고 안에 "안녕하세요!"라고 써줘.
```

Cowork가 권한을 묻거나 바로 만들어줍니다. 작업이 끝나면 메시지에 **파일 링크가 나타납니다** ("View your file" 같은 형태). 클릭하면 파일이 열립니다.

```
hello.txt 파일을 한 줄로 요약해줘.
```

이번엔 Read 권한을 묻고, 내용을 읽어 요약해줍니다.

### 3.3 Cowork가 일반 채팅과 다른 점

다음 4가지를 처음에 알아두면 헷갈리지 않습니다.

#### (1) 폴더 연결이 핵심

폴더를 연결하지 않으면 Cowork는 **임시 작업 공간**에서만 일합니다. 결과 파일이 다음에 사라질 수 있습니다. **항상 작업 시작 시 폴더 연결**부터 하세요.

#### (2) 파일 링크가 자주 나옴

작업이 끝나면 메시지에 **"View your file"** 같은 링크가 들어 있습니다. 그게 결과물입니다. 잊지 말고 클릭해보세요.

#### (3) 권한을 묻는 일이 많음

처음엔 매번 "이 작업 해도 될까요?"라고 묻습니다. 안전을 위해서입니다. 자주 쓰는 동작은 **"이 세션 동안 항상 허용"** 옵션을 선택할 수 있습니다.

#### (4) AskUserQuestion (선택지 질문)

Cowork는 가끔 **객관식 질문 카드**를 보여줍니다. "이 보고서 분량은? ① 짧게 ② 보통 ③ 자세히" 같은. 의도를 명확히 하기 위해서니 부담 없이 답변하시면 됩니다.

### 3.4 Cowork 사이드바 위젯들

대화 화면 옆에 다음 위젯들이 자주 등장합니다.

| 위젯 | 의미 | 활용 |
| --- | --- | --- |
| **TodoList** | Claude가 작업을 단계별로 추적 | 길어진 작업의 진행률 보기 |
| **Artifact** | Claude가 만든 라이브 페이지 (11장 참고) | 매일 다시 열어보는 대시보드 |
| **Scheduled Tasks** | 예약된 자동 작업 (12장 참고) | 매주 자동 보고서 생성 등 |

## 4. Claude Code로 시작하기 (터미널이 익숙한 분)

> 이 챕터는 **터미널 사용 경험이 있는 분**을 위한 것입니다. 터미널이 처음이라면 3장(Cowork)부터 시작하시는 것이 훨씬 부드럽습니다.

### 4.1 무엇을 설치하나요?

Claude Code는 **터미널**(검은 화면)에서 실행되는 프로그램입니다.

- Mac: 돋보기(🔍) → "터미널" → Enter
- Windows: 시작 → "PowerShell" → Enter

### 4.2 단계별 설치

#### 1단계 — Node.js 설치 (없으면)

1. https://nodejs.org → **LTS** 버전 다운로드
2. 안내대로 설치
3. 터미널 새로 열고 확인:
   ```
   node --version
   ```

#### 2단계 — Claude Code 설치

```
npm install -g @anthropic-ai/claude-code
```

#### 3단계 — 처음 실행

작업 폴더에서 실행합니다.

```
mkdir ~/claude-test
cd ~/claude-test
claude
```

처음엔 로그인을 요구합니다. 안내대로 Anthropic 계정으로 로그인하면 입력창(`>`)이 나타납니다.

### 4.3 첫 대화

```
hello.txt 파일을 만들고 안에 "안녕하세요!"라고 써줘.
```

Y로 권한을 승인하면 폴더에 파일이 만들어집니다.

### 4.4 자주 쓰는 명령어

| 명령어 | 용도 |
| --- | --- |
| `/help` | 도움말 |
| `/clear` | 대화 초기화 |
| `/init` | 현재 폴더에 기본 CLAUDE.md 자동 생성 |
| `/cost` | 토큰 사용량 |
| `/quit` | 종료 |

### 4.5 안 될 때 체크 리스트

| 문제 | 해결 |
| --- | --- |
| `command not found: claude` | 터미널 다시 열기, Node.js 설치 재확인 |
| `EACCES`, 권한 오류 | Mac/Linux는 `sudo npm install -g @anthropic-ai/claude-code` |
| 한글 깨짐 | Windows PowerShell이라면 `chcp 65001` |

## Part 2 — 똑똑한 비서로 만들기

## 5. CLAUDE.md / 메모리 — 내 프로젝트를 AI에게 소개하기

### 5.1 이게 왜 필요한가요?

매번 "우리 회사는 무슨 일을 하고", "어떤 도구를 쓰고", "어떤 규칙이 있고…"를 설명한다고 상상해보세요. 비효율적이죠.

`CLAUDE.md`는 **비서가 출근하면 가장 먼저 읽는 회사 소개서**입니다. 폴더에 이 파일이 있으면 Claude가 자동으로 읽고 시작합니다.

> **Cowork 사용자**: Cowork에서는 CLAUDE.md 외에도 **메모리 파일** 시스템을 쓸 수 있습니다(memory/ 폴더). 단순한 시작은 CLAUDE.md 한 개로 충분합니다.

### 5.2 어디에 만드나요?

작업 폴더의 루트에 `CLAUDE.md`라는 이름으로 만듭니다.

```
claude-test/
└── CLAUDE.md          ← 여기
```

### 5.3 단계별 따라하기

작업 폴더에서 Claude에게 다음을 입력합니다.

```
이 폴더에 CLAUDE.md 파일을 만들고 다음 내용을 넣어줘:
- 모든 답변은 한국어로
- 결론을 먼저, 이유는 그 다음
- 파일을 만들거나 수정할 때마다 어디에/무엇을 변경했는지 알려주기
```

Y로 승인. 폴더에 `CLAUDE.md`가 생깁니다.

### 5.4 효과 확인

새 대화를 시작해 다음을 물어보세요.

```
지금 우리 프로젝트에서 지켜야 할 규칙이 뭐야?
```

CLAUDE.md에 적은 규칙을 그대로 알려주면 성공입니다.

### 5.5 무엇을 적으면 좋을까?

```markdown
# 우리 팀 프로젝트

## 프로젝트 개요
- 무엇을 만드는 프로젝트인지 1-2문장
- 누구를 위한 것인지

## 답변 스타일

- 결론 먼저, 그다음 이유

## 자주 쓰는 폴더

- 회사 외부 사이트로 데이터 보내지 않기
```

## 6. Skill — AI에게 매뉴얼을 쥐여주기

### 6.1 이게 왜 필요한가요?

CLAUDE.md는 "**프로젝트 전체에 항상 적용되는 규칙**"입니다.
Skill은 "**특정 상황에서만 꺼내 보는 매뉴얼**"입니다.

예: "회의록 작성할 때만 쓰는 양식", "고객 메일 톤", "월간 보고서 구조" 등.

### 6.2 비유로 이해하기

| | 비유 |
| --- | --- |
| CLAUDE.md | 신입사원 OT 자료 (모두가 항상 알아야 함) |
| Skill | 업무 매뉴얼 (해당 업무할 때만 펼침) |

### 6.3 Skill 폴더 구조

```
프로젝트폴더/
└── .claude/
    └── skills/
        └── 회의록-작성/
            └── SKILL.md
```

> **Cowork 사용자 팁**: Cowork에서도 동일한 폴더 구조를 인식합니다. 또한 **Plugin**(14장)으로 묶어 여러 Skill을 한 번에 설치할 수도 있습니다.

### 6.4 단계별 따라하기 — 첫 Skill 만들기

#### 1단계 — Claude에게 폴더와 파일 만들어달라고 하기

```
.claude/skills/회의록-작성 폴더를 만들고, 그 안에 SKILL.md 파일을 만들어줘.
파일 내용은 다음과 같이 시작해줘:

---
name: 회의록-작성
description: |
  회의록 작성, 정리, 요약 시 사용. 트리거 키워드: "회의록", "미팅 노트",
  "주간 회의", "회의 정리", "오늘 회의", "결정사항 정리".
---

# 회의록 작성 매뉴얼

## 양식
- 일시:
- 참석자:
- 안건:
- 논의 요약:
- 결정 사항:
- TODO (담당자/기한):

## 작성 원칙
- 발언자 이름은 닉네임 그대로
- 결정 사항은 동사로 시작 (예: "X를 도입한다")
- TODO는 담당자와 기한 둘 다 기재
```

#### 2단계 — 사용해보기

```
오늘 마케팅 팀과 회의했어. 안건은 5월 캠페인 예산이었고,
민지가 데이터 분석 맡고 다음 주 화요일까지, 준호가 디자인 시안 다음 주 금요일까지로 결정했어.
회의록으로 정리해줘.
```

Claude가 자동으로 위 Skill을 읽고 정해둔 양식대로 정리해줍니다.

### 6.5 description 잘 쓰는 법 (가장 중요!)

Skill을 만들었는데도 호출이 안 된다면 99% description 문제입니다.

#### 나쁜 예 ❌
```
description: 회의록을 잘 만들어줍니다.
```

#### 좋은 예 ✅
```
description: |
  회의록 작성, 정리, 요약, 미팅 결과 정리, 회의 결정 사항 정리할 때 사용.
  트리거 키워드: "회의록", "미팅 노트", "meeting notes", "minutes",
  "주간 회의", "회의 정리", "오늘 회의", "결정사항 정리".
```

규칙: **사용자가 실제로 쓸 표현을 다양하게**(한·영 양쪽 권장).

### 6.6 비개발 업무에 좋은 Skill 예시

| Skill 이름 | 언제 쓰는가 | 핵심 내용 |
| --- | --- | --- |
| 고객-이메일-작성 | 고객 응대 메일 | 인사말, 톤, 마무리 문구 |
| 보고서-요약 | 긴 문서 요약 | "결론 → 근거 → 다음 액션" 순서 |
| 데이터-표-만들기 | 숫자 표 만들기 | 헤더 규칙, 단위 표기 |
| 사내공지-작성 | 공지 사항 | 제목 형식, 본문 길이 |
| 발표자료-목차 | 발표 슬라이드 | 표지 → 문제 → 해결 → 효과 → Q&A |

### 6.7 체크리스트

- [ ] 폴더는 `.claude/skills/<이름>/`?
- [ ] 파일명은 정확히 `SKILL.md`? (대소문자)
- [ ] `name`이 폴더명과 같은가?
- [ ] description에 트리거 키워드 5개 이상?
- [ ] 본문은 양식·절차 위주? (설명 줄글보다 체크리스트)

## 7. Subagent — AI에게 전문가 동료를 만들어주기

### 7.1 이게 왜 필요한가요?

지금까지의 Claude는 **만능 비서 한 명**입니다. Subagent는 **한 가지 일에 특화된 전문가**입니다.

### 7.2 비유

| 사람 | Skill | Subagent |
| --- | --- | --- |
| 비유 | 책장에 꽂힌 매뉴얼 | 옆 자리에 앉은 동료 |
| 사용 방식 | 메인이 책 펼쳐 보고 직접 함 | 메인이 "이건 네가 해" 하고 위임 |
| 머리(컨텍스트) | 메인과 같은 머리 | 자기만의 머리 (생각 섞이지 않음) |
| 권한 | 메인과 같음 | 따로 정할 수 있음 |

### 7.3 폴더 구조

```
프로젝트폴더/
└── .claude/
    └── agents/
        └── 보고서-검토관.md     ← 폴더 아닌 파일 하나
```

> **Cowork 사용자**: Cowork에서는 메인 비서가 자동으로 적절한 Subagent에게 위임합니다. UI 차이는 거의 없고, 결과만 깔끔하게 사용자에게 보여집니다.

### 7.4 단계별 따라하기

#### 1단계 — 파일 만들기

```
.claude/agents/보고서-검토관.md 파일을 만들어줘. 내용은:

---
name: 보고서-검토관
description: |
  보고서·문서·글의 오탈자·문장·구조를 검토할 때 사용.
  트리거: "검토", "리뷰", "교정", "오탈자", "문장 봐줘", "다듬어줘".
tools: Read, Grep, Glob
---

당신은 한국어 보고서 검토 전문가입니다.

# 책임
- 주어진 글을 읽고 개선점을 찾습니다.
- 직접 글을 수정하지는 않습니다. 코멘트만 작성합니다.

# 검토 항목
1. 오탈자, 띄어쓰기
2. 어색한 문장, 중복 표현
3. 논리 흐름 (결론이 먼저 나오는가?)
4. 숫자·날짜 표기 일관성

# 보고 형식
## 🔴 꼭 고쳐야 할 점
- [위치] 문제 → 수정 제안

## 🟡 검토해볼 점
- [위치] 의견

## 🟢 잘 쓴 부분
- [위치] 칭찬

# 원칙
- 5개 넘으면 가장 중요한 5개만 고른다.
- "왜 그렇게 고쳐야 하는지" 한 줄로 같이 적는다.
```

> 💡 `tools: Read, Grep, Glob`은 **읽기 전용**으로 권한을 제한한 것입니다. 검토자가 실수로 글을 수정하지 못하게 합니다.

#### 2단계 — 사용해보기

```
report.md 좀 검토해줘.
```

자동으로 보고서-검토관에게 위임됩니다. 명시적으로도 가능:

```
보고서-검토관 에이전트로 report.md를 검토해줘.
```

### 7.5 비개발 업무 Subagent 예시

| Subagent | 책임 | 권한 |
| --- | --- | --- |
| 보고서-검토관 | 글 검토, 코멘트 | 읽기만 |
| 데이터-요약가 | 큰 표/엑셀 핵심 요약 | 읽기 + 명령 실행 |
| 회의록-정리가 | 음성 기록 → 양식 회의록 | 읽기 + 쓰기 |
| 자료-수집가 | 주제 받아 자료 수집 → 표 정리 | 검색 + 읽기 |
| 영문-번역가 | 한 ↔ 영 자연스러운 번역 | 읽기 + 쓰기 |

### 7.6 Skill과 Subagent — 어느 쪽?

| 상황 | 권장 |
| --- | --- |
| "이 양식 그대로 따라줘" | **Skill** |
| 매번 양식이 비슷한 작업 | **Skill** |
| 검토자/작성자처럼 역할 분리 | **Subagent** |
| 큰 작업이 메인 대화를 잡아먹음 | **Subagent** |
| 권한 분리 (읽기만 등) | **Subagent** |

## 8. Slash Command — 자주 쓰는 말 단축키 만들기

### 8.1 이게 왜 필요한가요?

매주 똑같은 긴 지시를 입력하기 번거롭습니다. Slash Command는 그 긴 지시를 `/주간요약` 한 줄로 줄여줍니다.

### 8.2 폴더 구조

```
프로젝트폴더/
└── .claude/
    └── commands/
        └── 주간요약.md
```

> **Cowork 사용자**: Cowork에서는 Skill을 곧바로 `/skill-name` 형태로 호출할 수 있어, 별도의 Slash Command 폴더 없이도 같은 효과를 얻을 수 있습니다. Claude Code는 commands 폴더가 정석입니다.

### 8.3 단계별 따라하기

#### 1단계 — 파일 만들기

```markdown
---
description: 지난 주 회의록을 모아 주간 요약 보고서로 정리
---

지난 주(월요일~일요일)에 작성된 모든 회의록 파일을 찾아줘.

## 절차
1. /reports 폴더에서 회의록 파일 검색
2. 각 회의록에서 "결정 사항"과 "TODO" 섹션만 추출
3. 다음 양식으로 통합 보고서 작성:

## 📅 주간 요약 ({날짜 범위})

### 🎯 주요 결정
- ...

### ✅ 진행 중인 TODO
- (담당자) 항목 — 마감일

### 🔄 다음 주 핵심
- ...
```

#### 2단계 — 사용해보기

```
/주간요약
```

#### 3단계 — 인자 받기 (조금 더 고급)

```markdown
---
description: 지정한 기간의 회의록을 주간 요약으로
argument-hint: <시작일> <종료일>
---

$ARGUMENTS 기간의 회의록을 모아 주간 요약을 작성해줘. 절차는…
```

사용:

```
/주간요약 2026-05-01 2026-05-07
```

### 8.4 비개발 업무 Slash Command 예시

| 명령어 | 용도 |
| --- | --- |
| `/일일정리` | 오늘 받은 메일·메시지 핵심만 정리 |
| `/회신초안` | 받은 이메일 답장 초안 |
| `/보고서요약` | 긴 문서 한 장 요약 |
| `/번역` | 한 ↔ 영 자동 번역 |
| `/표만들기` | 텍스트를 비교 표로 변환 |

### 8.5 세 도구 한눈에 비교

| 누가 부르나? | 어떻게 부르나? | 적합한 경우 |
| --- | --- | --- |
| **Skill** — Claude가 알아서 | 사용자가 적당한 단어 입력 | 항상 따라야 할 양식 |
| **Subagent** — 메인이 부름 | 자동 위임 또는 명시 호출 | 역할/권한 분리 |
| **Slash Command** — 사람이 직접 | `/명령어` 입력 | 같은 긴 지시 반복 |

## 9. Hook — 자동으로 실행되는 약속

### 9.1 무엇인가요?

**"이런 일이 일어나면 자동으로 X 실행"** 하는 자동 약속입니다. 사무실 출입문에 동작 센서를 다는 것과 비슷합니다.

### 9.2 비개발자에게는 우선순위 낮음

솔직히, **Hook은 Skill/Subagent/Slash Command를 충분히 익힌 다음**에 봐도 늦지 않습니다. 처음부터 Hook부터 만들면 디버깅이 어려울 수 있습니다.

> **Cowork 사용자**: Cowork에서는 Hook 설정이 직접 노출되지 않을 수 있습니다. 자주 쓰는 자동화는 **Scheduled Task**(12장)로 해결되는 경우가 많아 Hook을 따로 다룰 필요가 거의 없습니다.

### 9.3 가장 무난한 Hook — 작업 끝나면 알림 (Mac)

`.claude/settings.json`:

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"작업 완료!\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}
```

이제 작업 끝마다 시스템 알림이 뜹니다.

## Part 3 — Cowork만의 기능

> 이 Part는 **Cowork 사용자만** 해당됩니다. Claude Code만 쓰신다면 15장으로 건너뛰셔도 됩니다.

## 10. 폴더 연결 — Cowork에 내 작업 공간 알려주기

### 10.1 왜 폴더 연결이 중요한가?

Cowork는 기본적으로 **임시 작업 공간**(outputs 폴더)에서 일합니다. 임시 공간은 세션이 끝나면 사라질 수 있습니다.

폴더를 연결하면, Cowork가 **내 컴퓨터의 실제 폴더에서 일하므로** 결과 파일이 영구 보존됩니다.

### 10.2 단계별 따라하기

#### 1단계 — 작업 폴더 정하기

바탕화면이나 문서 폴더 안에 새 폴더를 만듭니다. 예: `claude-work`.

#### 2단계 — Cowork에 알려주기

```
바탕화면의 claude-work 폴더에서 작업할게.
```

또는 화면 어딘가의 **"폴더 선택"** 버튼을 클릭.

#### 3단계 — 권한 부여

OS가 "Claude가 이 폴더에 접근할 수 있게 허용하시겠습니까?"라고 물을 수 있습니다. **허용**.

#### 4단계 — 확인

```
지금 어떤 폴더에서 작업 중이야?
```

연결한 폴더 경로를 알려주면 성공입니다.

### 10.3 폴더 안의 파일 다루기

```
이 폴더에 있는 PDF들 제목만 알려줘.
```

```
2025-04로 시작하는 파일들을 archive 폴더로 옮겨줘.
```

```
회의록.docx 파일을 읽고 핵심 결정 사항만 새 파일로 정리해줘.
```

### 10.4 주의사항

- 중요한 파일이 있는 폴더를 연결할 때는 **백업**부터 해두세요.
- CLAUDE.md에 "**X 폴더는 절대 수정하지 말 것**"을 적어두면 안전망이 됩니다.

## 11. Artifact — 다시 열어보는 라이브 페이지

### 11.1 무엇인가요?

Artifact는 **Cowork가 만든 작은 웹페이지**입니다. 일반 채팅 답변과 다른 점은 다음과 같습니다.

- **저장**됩니다. 며칠 뒤에도 다시 열어볼 수 있습니다.
- **자동 새로고침**됩니다. 슬랙·노션 같은 외부 데이터를 매번 새로 가져옵니다.
- **인터랙션**이 됩니다. 버튼 누르고 필터 걸고 정렬할 수 있습니다.

### 11.2 비유로 이해하기

- 일반 답변 = **출력된 종이 한 장** (그 시점의 스냅샷)
- Artifact = **내 책상에 둔 모니터** (계속 데이터가 갱신됨)

### 11.3 어떤 경우에 Artifact가 좋은가?

| 상황 | 일반 답변 | Artifact |
| --- | --- | --- |
| "이번 주 매출이 얼마야?" (단발성) | ✅ | ❌ |
| "매일 아침 확인할 매출 대시보드" | ❌ | ✅ |
| "이 자료 한 번 요약해줘" | ✅ | ❌ |
| "내 할 일 페이지 만들어줘" (반복) | ❌ | ✅ |
| "이 PDF 핵심만 알려줘" | ✅ | ❌ |
| "팀 채용 파이프라인 현황 페이지" | ❌ | ✅ |

기준: **"이걸 다음에도 다시 열어보고 싶은가?"** + **"데이터가 시간에 따라 바뀌는가?"** 둘 다 ✅이면 Artifact.

### 11.4 단계별 따라하기 — 첫 Artifact

가장 직관적인 예시: **할 일 대시보드**.

#### 1단계 — Cowork에 요청

```
오늘부터 매일 아침에 열어볼 할 일 대시보드 페이지를 만들어줘.
- 오늘 해야 할 일 (체크박스)
- 진행 중인 일
- 이번 주에 마감인 일
가장 위에는 인사말과 오늘 날짜가 보이게 해줘.
```

#### 2단계 — Cowork가 Artifact 생성

Cowork가 작은 HTML 페이지를 만들어 사이드 위젯으로 띄웁니다. 위에 **"Artifact"** 같은 표시가 붙습니다.

#### 3단계 — 다시 열기

다음 날, Cowork를 켜면 사이드바의 **Artifacts 목록**에 어제 만든 페이지가 그대로 있습니다. 클릭하면 같은 페이지가 다시 열립니다.

#### 4단계 — 새로고침

페이지 위쪽에 **"Reload"** 버튼이 있습니다. 누르면 데이터가 다시 불러집니다(예: 오늘 날짜로 갱신, 새 슬랙 메시지 반영 등).

### 11.5 진짜 강력한 활용 — Connector와 결합

Artifact가 진짜 빛나는 건 **Connector**(13장)와 결합할 때입니다.

```
슬랙의 #프로젝트-A 채널에서 어제 오간 메시지를 핵심만 묶어주는 페이지를 만들어줘.
매일 아침 열면 자동으로 어제 분이 갱신되게.
```

이러면 매일 아침 그 페이지만 열면 어제 슬랙 핵심이 자동으로 보입니다.

### 11.6 Artifact 활용 아이디어

| 페이지 | 데이터 출처 |
| --- | --- |
| 매일 아침 슬랙 핵심 | Slack Connector |
| 노션 미해결 페이지 목록 | Notion Connector |
| 구글 캘린더 오늘 일정 + 준비물 | Google Calendar |
| 팀 OKR 진척도 | Asana / Jira / Linear |
| 받은 메일 분류 (긴급/일반/뉴스레터) | Gmail Connector |

## 12. Scheduled Task — 정해진 시간에 자동 실행

### 12.1 무엇인가요?

**"매주 금요일 오후 5시에 자동으로 X를 해줘"** 같은 예약 작업입니다. 알람 시계와 비슷합니다.

### 12.2 어떤 경우에 좋은가?

- 매일/매주 **반복되는** 정리 작업
- 사람이 **잊어버리기 쉬운** 작업 (월말 보고, 분기 회고)
- **시간대가 정해진** 작업 (퇴근 전 정리, 아침 브리핑)

### 12.3 단계별 따라하기

#### 1단계 — Cowork에 요청

```
매주 금요일 오후 5시에 자동으로 다음을 해줘:
1. /reports 폴더의 이번 주 회의록을 모두 모아
2. 결정 사항과 TODO를 추출하고
3. weekly-report-{날짜}.md 파일로 저장
4. 보고서-검토관 에이전트에게 검토받아서 코멘트도 같이 첨부

이걸 매주 같은 시간에 실행되도록 예약 작업으로 만들어줘.
```

#### 2단계 — Cowork가 예약 등록

Cowork가 **Scheduled Tasks**에 새 항목을 추가합니다. 사이드바에 작업 이름·다음 실행 시간이 표시됩니다.

#### 3단계 — 결과 확인

매주 금요일 오후 5시에 자동 실행됩니다. 결과 파일이 폴더에 생기고, 알림으로 "주간 보고서 완성"을 알려줍니다.

### 12.4 일회성으로도 가능

**1시간 뒤에** 같은 일회성 예약도 됩니다.

```
1시간 뒤에 이 회의록을 5줄로 요약해서 알려줘.
```

### 12.5 Scheduled Task vs 사람 알림

| 비교 | 사람의 알람 시계 | Scheduled Task |
| --- | --- | --- |
| 알림만 줌 | ✅ | ✅ |
| 자동으로 작업까지 끝냄 | ❌ | ✅ |
| 결과를 정리해서 보여줌 | ❌ | ✅ |

### 12.6 활용 아이디어

| 주기 | 작업 |
| --- | --- |
| 매일 오전 9시 | 어제 슬랙 핵심 + 오늘 캘린더 요약 |
| 매주 금요일 17시 | 주간 보고서 자동 작성 |
| 매월 말일 | 월간 회고 초안 (KPT 양식) |
| 분기말 | 분기 OKR 진척도 보고서 |
| 1시간 후 | 회의 끝났는지 확인 + 회의록 정리 |

## 13. Connector — 슬랙·노션·구글을 Claude와 연결

### 13.1 무엇인가요?

Connector는 **Claude를 외부 서비스에 연결해주는 다리**입니다. 한번 연결해두면 Claude가 그 서비스의 데이터를 직접 읽고/쓸 수 있습니다.

### 13.2 비유

- Connector 없는 Claude = **사무실에만 있는 비서** (외부 도구 못 봄)
- Connector 연결된 Claude = **회사 시스템 다 접근 가능한 비서** (슬랙·노션·구글까지)

### 13.3 자주 쓰는 Connector 종류

| 카테고리 | 서비스 |
| --- | --- |
| 메신저 | Slack, MS Teams |
| 문서/노트 | Notion, Confluence, Google Docs |
| 이메일/캘린더 | Gmail, Google Calendar, Outlook |
| 프로젝트 관리 | Asana, Jira, Linear, ClickUp, Monday |
| 파일 저장 | Google Drive, OneDrive, Dropbox, Box |
| 디자인 | Figma |
| 개발 | GitHub, GitLab |
| 데이터 | Amplitude, BigQuery, Snowflake |
| 고객 지원 | Intercom, Zendesk |

### 13.4 단계별 연결하기 (예: Slack)

#### 1단계 — Cowork에 요청

```
슬랙 연결해줘.
```

또는 Cowork 메뉴/설정에서 **Connectors** → **Slack** 찾기.

#### 2단계 — 인증

Cowork가 슬랙 로그인 페이지를 띄웁니다. 본인 슬랙 계정으로 로그인하고 **"권한 허용"** 클릭.

> 💡 **권한 범위 확인**: 무엇에 접근하는지 꼭 확인하세요. "메시지 읽기"만 필요한지, "쓰기"도 필요한지에 따라 권한을 좁힐 수 있다면 좁히는 것이 안전합니다.

#### 3단계 — 사용해보기

```
어제 #general 채널에서 오간 메시지 핵심만 요약해줘.
```

```
다음 주 회의 제안을 #design 채널에 메시지로 보낼 거야. 초안만 작성해줘. (보내지는 마)
```

> ⚠️ Cowork는 **메시지 전송, 파일 삭제 같은 외부 영향이 있는 작업** 직전에 항상 사용자에게 권한을 묻습니다. "보내도 될까요?" 단계에서 확인하고 진행하세요.

### 13.5 Connector + Skill + Artifact 조합 예시

세 가지를 합치면 강력해집니다.

**시나리오**: 매일 아침 어제의 슬랙 핵심을 페이지로 보기.

1. **Connector**: Slack 연결
2. **Skill**: "슬랙 메시지 요약 시 출처 채널·발신자 표기" 같은 매뉴얼
3. **Artifact**: 매일 새로고침되는 페이지로 표시
4. **Scheduled Task**: 매일 아침 9시에 자동 갱신

```
슬랙의 #프로젝트-A, #design 두 채널에서 어제 오간 메시지를 다음처럼 보여주는 페이지를 만들어줘:
- 채널별 핵심 3가지
- 결정된 사항이 있다면 별도 표시
- 누가 발언했는지 함께
매일 아침 9시에 어제 분 데이터로 자동 새로고침되게.
```

이 한 번의 요청으로 **Connector + Skill + Artifact + Scheduled Task** 4가지가 동시에 설정됩니다.

### 13.6 안전 수칙

- 회사 데이터에 접근하는 Connector는 **회사 정책에 어긋나지 않는지** 먼저 확인
- 시크릿(API 키 등)은 Skill이나 메모에 절대 적지 않기
- 메시지 전송, 파일 삭제는 항상 **사람의 최종 확인**

## 14. Plugin — 묶음으로 설치하는 도구 세트

### 14.1 무엇인가요?

Plugin은 **Skill + Subagent + Slash Command + Connector를 한 묶음으로 설치하는 패키지**입니다.

비유: 새 회사에 출근했을 때 **"우리 팀 신입 키트"**로 USB 한 개 받는 셈입니다. 그 안에 매뉴얼·연락처·도구가 다 들어있습니다.

### 14.2 어떤 Plugin이 있나?

대표 예시:

| Plugin | 안에 무엇이 들었나 |
| --- | --- |
| **Productivity** | 할 일 관리 Skill, TASKS.md 시스템, 메모리 관리 |
| **Engineering** | 코드 리뷰어, 디버그 가이드, 시스템 설계 매뉴얼 |
| **Design** | UX 카피 매뉴얼, 접근성 검토관, 디자인 시스템 점검 |
| **Data** | SQL 작성, 데이터 시각화, 통계 분석 도구 |
| **Anthropic Skills** | docx/pptx/xlsx/pdf 작성 매뉴얼, MCP 빌더 |

### 14.3 한 단계씩 따라하기 — Plugin 설치

#### 1단계 — Cowork에 요청

```
내가 마케팅 일을 하는데 도움될 만한 Plugin이 있을까? 추천해줘.
```

또는 Cowork 메뉴의 **Plugins/Marketplace**에서 직접 검색.

#### 2단계 — 추천 Plugin 보기

Cowork가 내 직무·작업에 맞는 Plugin 후보를 보여줍니다. AskUserQuestion 카드로 **선택지**가 나올 수 있습니다.

#### 3단계 — 설치

원하는 Plugin을 선택. 권한 확인 후 **"설치"**.

#### 4단계 — 사용

설치 직후부터 그 Plugin 안의 Skill·Subagent들이 자동으로 동원됩니다. 별도 설정 없습니다.

```
내 직무에 맞는 Skill 목록 알려줘.
```

설치된 Plugin이 제공하는 Skill들이 보입니다.

### 14.4 첫 Plugin 추천 (비개발자)

| 직무/상황 | 권장 Plugin |
| --- | --- |
| 글·문서 위주 업무 | Anthropic Skills (docx/pptx/xlsx/pdf) |
| 할 일·일정 관리가 핵심 | Productivity |
| 디자인 검토·UX | Design |
| 데이터 보고가 자주 있음 | Data |

### 14.5 Plugin vs 직접 만들기

처음에는 **이미 있는 Plugin부터 설치**하시고, 부족한 부분만 직접 Skill로 추가하는 것이 효율적입니다. 모든 걸 직접 만들면 시간이 오래 걸립니다.

## Part 4 — 실전과 팀 운영

## 15. 실전: 여러 도구를 합쳐서 쓰기 (오케스트레이션)

### 15.1 오케스트레이션이 뭔가요?

**오케스트라**는 여러 악기를 지휘자가 모아 한 곡을 연주하는 것입니다.
**오케스트레이션**은 Skill·Subagent·Slash Command·Connector·Artifact·Scheduled Task를 한 번에 모아 한 작업을 끝내는 것입니다.

### 15.2 시나리오 1 — 주간 보고서 풀 자동화 (Cowork)

**상황**: 매주 금요일 오후 5시에 다음을 자동으로.

1. 한 주간 슬랙 핵심
2. 노션의 미해결 작업
3. 구글 캘린더 다음 주 일정
4. 위 3가지를 합친 보고서
5. 보고서-검토관에게 검토 요청
6. 결과를 Artifact로 저장 (다음 주에 다시 볼 수 있게)

**준비물**

- **Connector**: Slack, Notion, Google Calendar
- **Skill**: 보고서-양식 (구조)
- **Subagent**: 보고서-검토관
- **Scheduled Task**: 매주 금 17:00
- **Artifact**: weekly-summary 페이지

**한 번의 요청으로 설정**

```
매주 금요일 17시에 자동 실행되는 작업을 만들어줘.

내용:
1. Slack의 #프로젝트-A, #design 채널에서 이번 주 오간 메시지를 핵심만 추출
2. Notion에서 이번 주 마감인데 아직 미해결인 페이지 목록
3. Google Calendar의 다음 주 일정과 준비 필요한 회의 목록
4. 이걸 합쳐서 보고서-양식 Skill 형식으로 작성
5. 보고서-검토관 에이전트에게 검토 받기
6. 검토 결과 반영본을 weekly-summary Artifact로 저장 (덮어쓰지 말고 매주 새 항목 추가)

확인용 미리보기를 한 번 보여주고 시작해줘.
```

이게 **진정한 오케스트레이션**입니다. 사람은 한 번 설정해두고, 매주 결과만 확인합니다.

### 15.3 시나리오 2 — 고객 응대 메일 워크플로우

**상황**: 받은 고객 메일을 빠르게 응대.

```
오늘 받은 미답변 고객 메일 목록을 보여주고, 각각에 대해 다음을 해줘:
1. 고객의 핵심 질문/요구 파악
2. 고객-이메일-작성 Skill 양식대로 답장 초안 작성
3. 다른 검토자에게 받지는 말고, 내가 확인할 수 있게 보여줘
4. 내가 확인한 뒤 "보내" 라고 하면 그때만 발송
```

**조합**: Gmail Connector + 고객-이메일-작성 Skill + 사람의 최종 확인.

### 15.4 핵심 패턴 4가지

| 패턴 | 흐름 | 예시 |
| --- | --- | --- |
| **순차 (Pipeline)** | A → B → C | 초안 → 검토 → 수정 → 저장 |
| **병렬 (Fan-out)** | A·B·C 동시에 | 번역가·검토관 동시 실행 |
| **검증 (Verification)** | 작성자 ≠ 검토자 | 메인이 쓰고 Subagent가 검토 |
| **분배 (Map-Reduce)** | 큰 일을 나눠 맡기기 | 회의록 50개를 3그룹으로 나눠 처리 |

### 15.5 실전 팁

1. **메인 비서는 가볍게**: 큰 일은 Subagent에 위임. 메인의 머리(컨텍스트)가 차면 답변 품질이 떨어집니다.
2. **결과 형식 못박기**: Subagent에 "이 형식으로 보고하세요"를 명시.
3. **사람 결정 지점은 멈추기**: 메일 발송, 파일 삭제 같은 영향 큰 작업은 자동화 끝까지 가지 말고 **사람 승인**에서 멈추기.
4. **Cowork의 AskUserQuestion 활용**: Claude가 객관식 선택지로 물으면 부담 없이 답하기.

## 16. 팀과 함께 쓰기

### 16.1 팀 도입 1주차 미니 로드맵

| 일차 | 할 일 | 도구 |
| --- | --- | --- |
| 1일 | 팀 폴더에 `CLAUDE.md` 작성 | 양쪽 공통 |
| 2-3일 | 가장 자주 쓰는 양식 1개 → Skill | 양쪽 공통 |
| 4-5일 | 검토관 Subagent 1개 도입 | 양쪽 공통 |
| 6일 | 자주 치는 긴 지시 → Slash Command | 양쪽 공통 |
| 7일 | (Cowork만) Connector 1개 + Artifact 1개 | Cowork |
| 둘째 주 | (Cowork만) Scheduled Task 1개 | Cowork |

**처음부터 Skill 10개, Subagent 5개 만들지 마세요.** 안 쓰는 매뉴얼이 쌓이면 헷갈립니다.

### 16.2 무엇을 공유, 무엇을 개인?

| 분류 | 팀 공유 | 개인 |
| --- | --- | --- |
| CLAUDE.md | 팀 공통 규칙 | 보통 만들지 않음 |
| Skills | 회의록·보고서 양식 | 개인 메모 양식 |
| Subagents | 보고서 검토관, 데이터 요약가 | 개인 학습용 |
| Slash Commands | `/주간보고`, `/일일정리` | `/번역` |
| Connector (Cowork) | 회사 슬랙·노션 | 개인 Gmail |
| Artifact (Cowork) | 팀 대시보드 | 개인 할 일 |
| Scheduled Task (Cowork) | 주간 자동 보고서 | 개인 알림 |

### 16.3 폴더 한눈에 보기 (Claude Code 기준)

```
우리팀-프로젝트/
├── CLAUDE.md
├── .claude/
│   ├── settings.json
│   ├── skills/
│   │   ├── 회의록-작성/SKILL.md
│   │   └── 보고서-양식/SKILL.md
│   ├── agents/
│   │   ├── 보고서-검토관.md
│   │   └── 데이터-요약가.md
│   └── commands/
│       ├── 주간보고.md
│       └── 일일정리.md
└── (실제 업무 파일들)
```

> **Cowork 운영 팁**: Cowork에서는 위 폴더 구조를 인식하면서, 추가로 **UI에서 관리하는 Plugins / Connectors / Scheduled Tasks**가 있습니다. 팀에 공유할 땐 Plugin 형태로 묶어서 내보내는 것이 가장 깔끔합니다.

### 16.4 깃허브로 공유

깃허브를 쓰는 팀이라면 `.claude/` 폴더를 함께 커밋해두면 됩니다. 새 멤버가 저장소를 받으면 같은 Skill·Subagent를 자동으로 사용 가능합니다.

깃을 안 쓴다면 폴더를 압축해 공유 드라이브에 올리는 것만으로 충분합니다.

## 17. 자주 묻는 질문 (FAQ)

### Q1. Cowork와 Claude Code 둘 다 설치해도 되나요?
**A**: 네. 같은 Anthropic 계정으로 양쪽 다 사용 가능합니다. 작업 성격에 맞춰 골라 쓰면 됩니다.

### Q2. Cowork에서 만든 Skill이 Claude Code에서도 동작하나요?
**A**: 폴더 구조(`.claude/skills/<이름>/SKILL.md`)가 같으므로 대부분 그대로 동작합니다. 폴더를 그대로 옮기면 됩니다.

### Q3. 만든 Skill이 호출이 안 돼요.
**A**: 99%는 description 문제입니다.
- 트리거 키워드를 더 다양하게 (한·영 양쪽)
- 파일명이 정확히 `SKILL.md`인지 (대소문자)
- Cowork라면 폴더 연결을 다시 확인

### Q4. Subagent 자동 위임이 안 돼요.
**A**: description이 약하면 메인이 위임을 결정 못합니다.
1. description에 트리거 키워드 보강
2. 명시적으로 호출:
   ```
   보고서-검토관 에이전트로 이 글 검토해줘.
   ```

### Q5. Cowork에서 Artifact를 다시 못 찾겠어요.
**A**: 사이드바의 **Artifacts** 또는 **저장된 페이지** 메뉴에서 목록을 볼 수 있습니다. 잊어버렸다면:
```
지금까지 만든 Artifact 목록 보여줘.
```

### Q6. Scheduled Task가 실행됐는지 어떻게 알아요?
**A**: 등록 시 알림 옵션을 켜두세요. 또는 "**최근 실행된 예약 작업 보여줘**"로 직접 확인 가능합니다.

### Q7. Connector 권한이 너무 광범위하게 보여요.
**A**: 가능한 한 **읽기 전용**으로 시작하세요. 쓰기 권한은 정말 필요할 때만. 회사 데이터라면 보안 담당자와 사전 협의 권장.

### Q8. Cowork에서 메시지가 너무 길거나 답변이 끊겨요.
**A**: 메인 컨텍스트가 가득 찼을 가능성. 다음 중 하나:
- 새 대화 시작
- 큰 작업은 Subagent에 위임
- 한 번에 모든 걸 시키지 말고 단계로

### Q9. 비용이 걱정돼요.
**A**: Cowork·Claude Code 모두 토큰 사용량에 따라 비용 발생.
- 일상 대화는 일반 Claude(claude.ai), 파일·자동화 필요할 때만 Cowork/Code
- Claude Code는 `/cost`로 사용량 확인
- Scheduled Task는 매번 비용이 발생하므로 빈도를 신중히 (매시간보다는 매일/매주)

### Q10. 실수로 파일을 망쳤으면?
**A**:
- 파일 수정 전 항상 권한을 묻습니다. 거기서 거절 가능
- 깃을 쓰면 `git checkout <파일>`로 복구
- 깃이 없으면 미리 백업 + CLAUDE.md에 "**X 폴더는 수정 금지**" 명시

### Q11. CLAUDE.md, Skill, Subagent, Plugin 다 만들어야 해요?
**A**: 절대 아닙니다. 권장 순서:

1. **CLAUDE.md** 1개 (5분)
2. (Cowork 사용자) 직무에 맞는 **Plugin 1개 설치**
3. 같은 작업 3번 반복했다면 **Skill 추가**
4. 검토자 분리 필요하면 **Subagent**
5. 같은 긴 지시 자주 친다면 **Slash Command**
6. 정해진 시간 자동 실행 (Cowork) → **Scheduled Task**
7. Hook은 정말 필요할 때만

### Q12. 영어와 한국어 키워드, 어느 쪽?
**A**: description에는 **양쪽 다** 권장.
```
description: 회의록 작성 시 사용. 트리거: "회의록", "미팅 노트", "meeting notes", "minutes", "회의 정리".
```

### Q13. 다른 사람의 Skill을 가져다 써도 되나요?
**A**: 네. 폴더 구조만 맞으면 그대로 복사 가능. 회사 공유 폴더의 Skill 모음이 있다면 적극 활용.

### Q14. Cowork에서 폴더 연결을 바꾸면 이전 폴더 작업은요?
**A**: 이전 폴더 안 파일은 그대로 남습니다. 다만 Cowork는 새로 연결된 폴더를 우선 사용합니다.

5. 셋을 합친 Slash Command 1개 만들기 → 이것만으로 매일 30분~1시간 절약

- [ ] Skill 1개를 처음부터 끝까지 만들어 동작 확인까지 했다
- [ ] Subagent 1개를 만들고 자동·명시 호출 모두 해봤다
- [ ] Slash Command 1개를 만들어 사용해봤다
- [ ] (Cowork) Connector 1개를 연결하고 데이터 가져와봤다
- [ ] (Cowork) Artifact 1개를 만들고 다음에 다시 열어봤다
- [ ] (Cowork) Scheduled Task 1개를 등록하고 자동 실행 확인
- [ ] 4가지 오케스트레이션 패턴 중 1개 이상 직접 사용해봤다

## 마무리

핵심을 한 문장으로 정리하면:

> **"AI 비서에게 회사 소개서(CLAUDE.md)를 주고, 매뉴얼(Skill)을 쥐여주고, 전문 동료(Subagent)를 붙여주고, 자주 쓰는 말은 단축키(Slash Command)로 만들고, 외부 도구는 Connector로 연결하고, 영구 페이지는 Artifact로, 자동 실행은 Scheduled Task로."**

처음부터 완벽한 시스템을 만들려고 하지 마세요. **딱 하나, 가장 자주 반복되는 작업 1개**를 골라 시작하세요. Cowork 사용자라면 우선 직무에 맞는 Plugin 1개 설치 + Skill 1개 추가가 가장 빠른 첫걸음입니다.

천천히, 한 단계씩 따라하시면 됩니다.
